# Python 环境配置智能体 — 功能规划与验证

## 问题定义

现有 AI 编程助手处理 Python 环境配置时采用串行试错：读取依赖声明，尝试安装，失败后修改再试，循环直至成功。这种模式在存在版本依赖冲突的项目上耗时且不可靠。

环境配置本质上是一个约束满足问题（CSP）：变量是各依赖包的版本号，约束来自 Python 版本、操作系统、C 扩展 ABI、CUDA 驱动等。失败信息比成功信息更有价值——每次安装失败都产生一个负约束（如"PyTorch 2.1.0 + NumPy 2.0 不兼容"），可以剪枝整个子空间。串行模式浪费了这些信息的即时价值。

本项目在已有 ReAct Agent 框架（MiniOpenClaw）之上，构建一个**并行自适应约束搜索（PACS）**子系统，将环境配置从串行试错转变为多路并行探测 + 约束传播驱动的搜索。

---

## 现有基础

MiniOpenClaw 提供了以下可直接复用的能力：

- ReAct 主循环：完成 tool call → 执行 → 结果反馈 → 下一步决策的闭环
- 内置工具集：read / write / bash / edit / grep / glob / web_fetch / task_list
- 技能系统：以 SKILL.md 形式注入领域知识到 system prompt
- MCP 客户端：可挂载外部 MCP server 扩展工具能力
- 安全层：命令沙箱、路径沙箱、SSRF 防护
- 评估框架：tool call 级别和 end-to-end 级别的评测任务定义

已有一个 `skills/codebase-guide/SKILL.md` 是关于 Python 项目环境配置的提示词说明文档，但它是静态文字指导，不具备实际的并行环境操作能力。

## 需要补全的功能

从"输入一个项目的依赖声明，输出一个可用的虚拟环境"的目标来看，以下功能不在现有框架中，需要新建：

### 功能一：并行环境池管理

现有条件：Agent 可通过 bash tool 调用系统命令，但这意味着串行执行 `pip install`，没有并行能力，没有环境隔离的抽象。

需要新增一个或一组工具，支持：
- 从 base Python 版本创建隔离环境实例
- 在同一时间点向多个环境并行下发安装任务
- 查询环境状态（空闲/运行中/已完成/失败）
- 清理环境释放资源

### 功能二：版本组合生成与调度

现有条件：没有依赖版本组合生成的能力；没有搜索策略。

需要新增一个组件（可作为 agent 调用的工具，或 Agent 遵循的 skill 流程）：
- 解析 `requirements.txt` / `pyproject.toml` 得到依赖约束图
- 根据当前知识库（已发现的冲突约束）枚举候选版本组合
- 对候选组合排序（信息增益、历史成功率）
- 将候选组合分配到空闲环境槽位

### 功能三：失败日志结构化解析

现有条件：bash tool 返回的是原始 stdout/stderr 文本，Agent 需要依赖 LLM 的理解能力去解析错误原因，不够可靠，且无法积累。

需要新增一个工具，将 pip 安装失败日志转化为结构化约束：
- 输入：安装日志文本 + 尝试的包-版本组合
- 输出：冲突约束列表（冲突双方、版本、错误原因、置信度）
- 内置覆盖 15 种以上常见 pip 失败模式的规则解析器

### 功能四：约束图与传播引擎

现有条件：没有任何约束表示或推理能力。

需要新增一个数据组件（本地文件或 SQLite 存储）：
- 约束图数据结构（节点：包+版本范围，边：冲突约束+置信度）
- 约束传播算法（传递推导：A 与 B 冲突 + B 与 C 冲突 → A 与 C 可能冲突）
- 剪枝接口（给定候选组合集合，返回过滤后的集合）
- 持久化（会话间复用已发现的约束）

### 功能五：整体搜索编排 skill

现有条件：Agent 不知道"并行搜索"这件事，需要教它如何组织上述能力完成目标。

需要编写一个 SKILL.md，描述 PACS 搜索策略的整体编排流程：
- 阶段一：广度优先采样——对关键决策点（Python 版本、核心框架版本）枚举正交组合并行验证
- 阶段二：约束传播——汇集失败结果，推导新约束，剪枝解空间
- 阶段三：精细化——在可行区域内微调次要依赖版本，锁定最终环境，输出 lock 文件

---

## 搜索策略

本系统采用**并行自适应约束搜索（PACS）**，分三波推进：

### 第一波：广度优先采样
快速验证关键决策点（Python 主版本、核心框架版本），并行启动 N 个环境测试正交组合，目标 2 分钟内定位"可行区域"。

### 第二波：约束传播
汇集第一波失败结果，更新全局约束图，推导新约束（如"PyTorch 2.0.x 与 NumPy 2.x 不兼容"），剪枝整个子空间。

### 第三波：深度精细化
在已验证的可行区域内，进行次要依赖版本微调，逐步安装并测试，锁定最终兼容组合。

## 输入-输出总览

### 系统级

输入：
- 项目依赖声明（`requirements.txt` / `pyproject.toml` 文件路径）
- Python 版本范围（可选，默认从项目推断）
- 并行度（可选，默认使用可用资源估算）

输出：
- 成功：可用虚拟环境路径 + 安装成功的依赖锁文件（等价于 `pip freeze > lock.txt` 的结果）
- 失败：失败原因说明 + 搜索过程中的约束图快照（已排除的组合及原因）

### 功能级

| 功能 | 输入 | 输出 |
|------|------|------|
| 并行环境池 | base Python 版本号, 槽位数 | 环境实例句柄列表（可用于下发任务） |
| 版本组合生成 | 依赖声明文件路径, 约束图 | 候选组合列表 `[{包名: 版本, ...}]` |
| 失败解析 | 安装日志文本, 尝试的版本组合 | 冲突约束列表 |
| 约束传播 | 已有约束 + 新约束 | 更新后的约束图 + 剪枝后的候选集 |
| 搜索编排 | 以上所有功能的组合调用 | 最终环境路径 + lock 文件 |

## 核查方式

### 整体功能核查

使用一个已知存在版本冲突的开源项目进行 end-to-end 测试：

步骤：
1. 准备一个测试项目，其 `requirements.txt` 中指定了互锁的依赖版本（如 `torch==2.0.1` + `numpy==2.0.0`，这对组合已知不兼容）
2. 通过 Agent 下发配置任务
3. 核查最终输出的是否为一个可激活的虚拟环境

通过标准：
- 运行 `python -c "import torch; import numpy"` 不报错
- 锁文件中保存了所有依赖的具体版本号
- 从下发到完成的时间，明显短于串行试错（可以用单独运行 `pip install` 做对照，但不作为硬性门槛）

### 功能级核查

**并行环境池**：
- 同时向 4 个环境下发不同的安装任务，用 `ps` 或系统进程监控验证 4 个 pip 进程同时运行
- 在一个环境中安装包后，其他环境的 import 不受影响

**版本组合生成**：
- 给定一个含 10 个依赖的简单项目，生成的结果中每个候选组合的包名与依赖声明一致
- 给定一个已知的不兼容组合（如 A==1 与 B==2 冲突），Planner 不应生成含该组合的候选

**失败解析**：
- 人工构造 10 条典型的 pip 失败日志（版本冲突、平台不兼容、Python 版本不匹配、系统库缺失等）
- 逐个输入解析器
- 每一条应产生至少一个结构化的约束，且冲突双方和错误类型与实际匹配

**约束传播**：
- 插入三条约束：A 1.x ↔ B 2.x，B 2.x ↔ C 3.x，C 3.x ↔ D 4.x
- 查询 A 1.x 相关的约束
- 应能推导出 A 1.x ↔ D 4.x

**搜索编排**：
- 可以在不重启 Agent 的情况下，通过指令触发两次独立的配置流程
- 第二次运行时如果项目中未引入新依赖，应能直接使用历史约束跳过部分探测步骤

### 量化度量

| 指标 | 定义 |
|------|------|
| **T_success** | 从启动到首个可用环境的时间 |
| **N_attempts** | 首次成功前的尝试次数 |
| **覆盖率** | 成功配置项目占尝试项目比例 |
| **资源消耗** | 平均 CPU 时间、峰值内存、磁盘占用 |
| **知识复用率** | 跨项目复用约束后，新项目首次尝试成功率 |
