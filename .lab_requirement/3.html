Day 3 下午 · 智能体评估 harness（M3）

一句话目标：先记录，后评估。今天你不写主循环（主循环是 D4 的事），而是搭一套评估 harness 的骨架——定义任务集、算指标、上 LLM-as-judge、记结构化轨迹、跑一次对照消融——并在样本轨迹（sample trajectories）上跑通。等 D4 有了真 agent，把它产出的轨迹喂进来即可，harness 一行不用改。 本次产出：eval/tasks.py（任务集 + 成功判据）、eval/metrics.py（成功率 / 步数 / token / JSON 合法率）、eval/judge.py（LLM-as-judge）、eval/tracer.py（JSONL 轨迹记录 + 回放），以及一份消融草稿（有/无 system-prompt 的成功率对比，呼应 D4 与最终展示）。 涉及的 starter 文件与标记：eval/tasks.py 的 # TODO[Day3]、eval/metrics.py 的 # TODO[Day3]，以及你新建的 eval/judge.py、eval/tracer.py。本日为本日构建目标（不单独计分，计入最终展示）。 与讲义的对应：任务集↔§9.1 端到端评测、指标↔§8 评估维度、judge↔§9.3、tracer↔§11 可观测性、消融↔§9.5 对照实验。建议边做边翻讲义对应小节。

步骤 1 · 定义评测任务集（文件：eval/tasks.py）

一次评测的起点是"我们到底要它做什么、怎么算做成了"。每个任务 = 名字 + 指令 + 成功判据（check）。判据要尽量程序化（讲义 §9.1）：给定一条轨迹记录，返回 True/False，不掺人的主观判断。今天先写 3～4 条样本任务，其中一条必须来自你组的领域。

  1. 打开 eval/tasks.py，定位 # TODO[Day3]。我们用一个 Task 结构，check 是一个函数（吃一条轨迹记录，吐布尔值）。填出下面的骨架： 
    from __future__ import annotations
from dataclasses import dataclass
from typing import Callable

# 一条“轨迹记录”长这样（步骤 2 会给出完整样本）：
#   {"task": "任务名", "steps": [ {tool_calls, raw, prompt_tokens, completion_tokens}, ... ],
#    "final": "agent 的最终自然语言答复"}
Trajectory = dict

@dataclass
class Task:
    name: str
    instruction: str                       # 给 agent 的指令
    check: Callable[[Trajectory], bool]    # 成功判据：吃一条轨迹，判成败

# ---- 成功判据（程序化优先）----
def _check_read_config(traj: Trajectory) -> bool:
    # 成功 = 期间调用过 read 且最终答复里报出了 timeout 的值
    used_read = any(
        tc["name"] == "read"
        for s in traj["steps"] for tc in s.get("tool_calls", [])
    )
    return used_read and "30" in traj.get("final", "")

def _check_list_dir(traj: Trajectory) -> bool:
    return any(
        tc["name"] == "bash" and "ls" in str(tc.get("arguments", {}))
        for s in traj["steps"] for tc in s.get("tool_calls", [])
    )

# TODO[Day3] 再补一条“你组领域”的任务判据（下面 _check_domain）
def _check_domain(traj: Trajectory) -> bool:
    # 例：法律组=答复含条款号；医疗组=答复含剂量；金融组=答复含金额……
    raise NotImplementedError("按你组领域实现：从 traj['final'] / steps 里判定是否达成")

SAMPLE_TASKS: list[Task] = [

步骤 2 · 成功率与效率指标（文件：eval/metrics.py）

讲义 §8 反复强调：报告一组指标，而非一个数字。今天实现四项——成功率（对不对）、步数 与 token 数（贵不贵，效率维度）、JSON 合法率（工具调用格式稳不稳）。为了在没有真 agent的情况下就能跑，我们内置一批样本轨迹记录，直接喂给指标函数。

  1. 打开 eval/metrics.py，定位 # TODO[Day3]，先把样本轨迹放进来（这就是"预录制的一次次 agent 运行"，D4 起会被真轨迹替换）： 
    from __future__ import annotations
import json, re
from typing import Any

TOOL_CALL_RE = re.compile(r"<tool_call>\s*(\{.*?\})\s*</tool_call>", re.DOTALL)

# 一条记录 = 一次任务运行留下的轨迹。steps 里每步含：模型这步请求的
# tool_calls、原始文本 raw（含 <tool_call>）、以及该步的 token 计数。
SAMPLE_RECORDS: list[dict[str, Any]] = [
    {"task": "read-config",
     "steps": [
         {"tool_calls": [{"name": "read", "arguments": {"path": "config.json"}}],
          "raw": '<tool_call>{"name":"read","arguments":{"path":"config.json"}}</tool_call>',
          "prompt_tokens": 310, "completion_tokens": 22},
     ],
     "final": "config.json 里 timeout = 30 秒。"},
    {"task": "list-dir",
     "steps": [
         {"tool_calls": [{"name": "bash", "arguments": {"command": "ls"}}],
          "raw": '<tool_call>{"name":"bash","arguments":{"command":"ls"}}</tool_call>',
          "prompt_tokens": 290, "completion_tokens": 18},
     ],
     "final": "当前目录有：main.py config.json README.md"},
    {"task": "read-config",          # 一条“失败/低质量”样本：JSON 被截断，且没报出值
     "steps": [
         {"tool_calls": [],
          "raw": '<tool_call>{"name":"read","arguments":{"path":',   # 坏 JSON
          "prompt_tokens": 305, "completion_tokens": 12},
         {"tool_calls": [], "raw": "我不确定 timeout 的值。",
          "prompt_tokens": 340, "completion_tokens": 15},
     ],
     "final": "我不确定 timeout 的值。"},
]
  2. 实现四个指标函数（填掉 # TODO[Day3]）： 
    def success_rate(tasks: list, records: list[dict]) -> float:
    """对每条 (task, trajectory) 记录跑 task.check，返回成功比例。"""
    by_name = {t.name: t for t in tasks}
    ok = 0
    for r in records:
        task = by_name.get(r["task"])
        if task and task.check(r):      # 复用步骤 1 的成功判据
            ok += 1
    return ok / max(len(records), 1)

def step_count(record: dict) -> int:
    return len(record["steps"])

def token_count(record: dict) -> int:
    return sum(s.get("prompt_tokens", 0) + s.get("completion_tokens", 0)
               for s in record["steps"])

def json_valid_rate(records: list[dict]) -> float:
    """从每步的 tool_calls（结构化）或 raw 里的 <tool_call>（文本）提取 JSON 并校验。"""
    total, ok = 0, 0
    for r in records:
        for s in r["steps"]:
            m = TOOL_CALL_RE.search(s.get("raw", ""))
            if not m:
                continue                # 这步没打算调工具，不计入
            total += 1
            try:
                json.loads(m.group(1)); ok += 1
            except json.JSONDecodeError:
                pass                     # 坏 JSON：计入分母、不计入分子
    return ok / max(total, 1)
  3. 写一个小驱动把指标算出来（可直接追加在文件末尾的 if __name__ == "__main__": 里，或新建 eval/run_metrics.py）： 
    if __name__ == "__main__":
    from eval.tasks import SAMPLE_TASKS
    recs = SAMPLE_RECORDS
    print("成功率        :", success_rate(SAMPLE_TASKS, recs))
    print("平均步数      :", sum(step_count(r) for r in recs) / len(recs))
    print("平均 token    :", sum(token_count(r) for r in recs) / len(recs))
    print("JSON 合法率   :", json_valid_rate(recs))
  4. ✅ 验证：在样本上算出四项指标： 
    python -m eval.metrics
    期望：能看到四行数字。成功率因为有一条失败样本应 < 1.0；JSON 合法率因为有一条坏 JSON 也应 < 1.0。能同时看到"对不对"和"贵不贵"两类维度就达标了（讲义 §8：别只报一个数字）。

步骤 3 · LLM-as-judge（文件：eval/judge.py）

有些任务没有程序化判据（"这段解释写得好不好"）。这时用一个模型按 rubric（评分量表）打分（讲义 §9.3）。今天复用 D2 打通的 backend/client.py 的 chat()，让 deepseek-v4-flash 给一个样本答复打分。judge 本身也有偏差，做完要能说清它们。

  1. 新建 eval/judge.py： 
    """LLM-as-judge：按固定 rubric 给一个答复打分（1-5）。"""
from __future__ import annotations
import re
from backend.client import DeepSeekBackend   # 复用 D2 的 chat()

RUBRIC = (
    "你是严格的评审。请按 1-5 分给【回答】打分：\n"
    "  5=完全正确且直接命中问题；3=部分正确或答非所问；1=错误或跑题。\n"
    "只依据【问题】判断【回答】，忽略回答的长度与措辞华丽程度。\n"
    "务必先写一行【理由】，再单独一行写【分数: X】（X 为 1-5 整数）。"
)

def judge(question: str, answer: str) -> dict:
    messages = [
        {"role": "system", "content": RUBRIC},
        {"role": "user", "content": f"【问题】{question}\n【回答】{answer}"},
    ]
    resp = DeepSeekBackend().chat(messages)      # deepseek-v4-flash，无 tools
    text = resp["content"]
    m = re.search(r"分数[:：]\s*([1-5])", text)   # 从“先理由后打分”里抠出分数
    score = int(m.group(1)) if m else None
    return {"score": score, "raw": text}

if __name__ == "__main__":
    q = "config.json 里 timeout 是多少？"
    for ans in ["timeout = 30 秒。", "我不太确定，可能是某个数吧。"]:
        r = judge(q, ans)
        print(f"答复={ans!r}  ->  分数={r['score']}")
        print("  judge 理由:", r["raw"].splitlines()[0])
  2. 注意两个刻意的设计（对应讲义 §9.3 的偏差缓解）： - rubric 固定并公开：评分标准写死在 RUBRIC 里，任何人重跑都用同一把尺子； - 先理由后打分：强制模型先给 【理由】 再给 【分数】，比直接吐一个数字更稳、也可供人工抽查。
  3. 想一想（写进笔记，呼应讲义 §9.3 的已知偏差）： - 冗长偏差：把上面第 2 个答复改得又长又客气但仍不给值，分数会不会被"哄高"？ - 一致性：同一条答复连跑 3 次，分数是否完全一样？（judge 有方差，单次打分不可尽信。） - 自我偏好 / 共享盲区：judge 和被评 agent 若是同一个模型，它发现不了自己的系统性错误——所以 judge 也需要被评估（抽样人工校准）。
  4. ✅ 验证： 
    python -m eval.judge
    期望：为两个样本答复各打印一个 1-5 的分数，正确的那条明显高于含糊的那条，并能看到 judge 给出的一行理由。（没配 DEEPSEEK_API_KEY 会报缺 key——这一步强依赖 API。）

步骤 4 · 可观测性：轨迹记录器（文件：eval/tracer.py）

讲义 §11 的口号：一个没有 trace 的 agent 无法被认真评估，也几乎无法被调试。 步骤 2 的样本轨迹是手写的；从 D4 起，真 agent 每跑一步都应自动发出一条结构化事件。这一步写一个极小的 tracer：把每步（工具调用、token）追加进 JSONL（一行一事件），再把它回放出来。

  1. 新建 eval/tracer.py： 
    """极小轨迹记录器：一步一行 JSON（JSONL），可回放。"""
from __future__ import annotations
import json, time
from pathlib import Path

class Tracer:
    def __init__(self, path: str):
        self.path = Path(path)
        self.path.write_text("", encoding="utf-8")   # 清空/新建

    def log_step(self, step: int, tool_calls: list, prompt_tokens: int,
                 completion_tokens: int, note: str = "") -> None:
        event = {"ts": round(time.time(), 3), "step": step,
                 "tool_calls": tool_calls, "note": note,
                 "prompt_tokens": prompt_tokens,
                 "completion_tokens": completion_tokens}
        with self.path.open("a", encoding="utf-8") as f:
            f.write(json.dumps(event, ensure_ascii=False) + "\n")

def replay(path: str) -> None:
    """把一条 JSONL 轨迹逐步打印出来（回放）。"""
    total_tok = 0
    for line in Path(path).read_text(encoding="utf-8").splitlines():
        e = json.loads(line)
        tok = e["prompt_tokens"] + e["completion_tokens"]
        total_tok += tok
        names = [tc["name"] for tc in e["tool_calls"]] or ["(无工具调用)"]
        print(f"  step {e['step']}: 调用 {names}  | 本步 {tok} tok  | {e['note']}")
    print(f"  —— 轨迹共 {total_tok} token")

if __name__ == "__main__":
    # 用步骤 2 的一条样本喂进来（模拟 D4 真 agent 逐步 log 的效果）
    from eval.metrics import SAMPLE_RECORDS
    rec = SAMPLE_RECORDS[0]
    tr = Tracer("eval/trace_sample.jsonl")
    for i, s in enumerate(rec["steps"]):
        tr.log_step(i, s.get("tool_calls", []),
                    s.get("prompt_tokens", 0), s.get("completion_tokens", 0),
                    note=s.get("raw", "")[:40])
    print(f"已写入 eval/trace_sample.jsonl（任务={rec['task']}）；回放：")
    replay("eval/trace_sample.jsonl")
  2. 记住设计原则：先记录，后评估。步骤 2 的四项指标，本质上都是对这种结构化 trace 的事后聚合——token_count 就是把每步的 token 加起来，json_valid_rate 就是扫每步的 tool_calls。D4 起你的主循环只要在每步调一次 tracer.log_step(...)，今天全部指标就能直接作用到真轨迹上。
  3. ✅ 验证： 
    python -m eval.tracer
    期望：生成 eval/trace_sample.jsonl（可 cat 看，一行一步），并在终端回放出逐步的工具调用与 token、末尾一行总 token。

步骤 5 · 跑一次对照（消融）on samples（沿用前四步文件）

讲义 §9.5：没有对照，任何"变好了"的说法都不可信。 这一步做一次最小消融——固定任务集，只改一个因素（有/无 system-prompt），比较两组样本轨迹的成功率，把结论记进消融草稿。这既是 D4（用消融量化上下文/压缩策略）的预演，也是最终展示要用的方法学。

  1. 准备两组样本轨迹（同一批任务、只差"有没有 system-prompt"）。新建 eval/ablation.py： 
    """最小消融：有/无 system-prompt 两组样本轨迹的成功率对比。"""
from eval.tasks import SAMPLE_TASKS
from eval.metrics import success_rate, token_count

# A 组：带 system-prompt（agent 被告知“需要时用 <tool_call> 调工具”）——都成功
GROUP_WITH_SYS = [
    {"task": "read-config",
     "steps": [{"tool_calls": [{"name": "read", "arguments": {"path": "config.json"}}],
                "raw": '<tool_call>{"name":"read","arguments":{"path":"config.json"}}</tool_call>',
                "prompt_tokens": 330, "completion_tokens": 22}],
     "final": "config.json 里 timeout = 30 秒。"},
    {"task": "list-dir",
     "steps": [{"tool_calls": [{"name": "bash", "arguments": {"command": "ls"}}],
                "raw": '<tool_call>{"name":"bash","arguments":{"command":"ls"}}</tool_call>',
                "prompt_tokens": 300, "completion_tokens": 18}],
     "final": "当前目录有：main.py config.json README.md"},
]
# B 组：无 system-prompt（agent 不知道工具约定，直接瞎答）——都失败
GROUP_NO_SYS = [
    {"task": "read-config",
     "steps": [{"tool_calls": [], "raw": "timeout 应该是个常见的默认值。",
                "prompt_tokens": 120, "completion_tokens": 14}],
     "final": "timeout 应该是个常见的默认值。"},
    {"task": "list-dir",
     "steps": [{"tool_calls": [], "raw": "你可以自己用 ls 看看。",
                "prompt_tokens": 110, "completion_tokens": 12}],
     "final": "你可以自己用 ls 看看。"},
]

def summarize(name, recs):
    sr = success_rate(SAMPLE_TASKS, recs)
    avg_tok = sum(token_count(r) for r in recs) / len(recs)
    print(f"{name:16s} 成功率={sr:.2f}  平均token={avg_tok:.0f}")
    return sr

if __name__ == "__main__":
    print("=== 消融：有/无 system-prompt ===")
    a = summarize("有 system-prompt", GROUP_WITH_SYS)
    b = summarize("无 system-prompt", GROUP_NO_SYS)
    print(f"结论：system-prompt 使成功率 {b:.2f} -> {a:.2f}（Δ={a-b:+.2f}）")
  2. 跑它： 
    python -m eval.ablation
  3. 把结论记进消融草稿（新建 eval/ablation_notes.md，最终展示会用到）。至少写清：改了哪一个变量、固定了什么、两组各自的成功率/token、以及能否归因。范例： 
    # 消融草稿（Day3 · 样本轨迹）
- 变量：system-prompt（有 / 无），其余（任务集、模型 deepseek-v4-flash）固定
- 结果：有=1.00 / 无=0.00；token：有更高（多了工具说明）
- 归因：无 system-prompt 时 agent 不知道 <tool_call> 约定 → 从不调工具 → 全失败
- 局限：样本量太小（各 2 条）、样本是构造的；D4 起用真轨迹、每组多次运行取均值再下结论
  4. ✅ 验证：python -m eval.ablation 打印出两组成功率与差值（有 system-prompt 一组明显更高），且 eval/ablation_notes.md 已写下"变量 / 固定项 / 结果 / 归因 / 局限"。

  • ☐ 你能解释：为什么 agent 评估不能只看"最终答案对不对"（讲义 §7：序列决策 + 非确定性 + 成本），必须同时看轨迹、方差、成本？相关评分见 ../00-课程总览/考核与评分总表.md §2。
  • ☐ 心里清楚接线：今天的 harness 在样本轨迹上跑通；D4 主循环建成后把真 agent 的轨迹换进来（harness 不改），并用步骤 5 的消融方法量化上下文压缩策略；D5 Skills、D6 安全会各自扩出新的评测维度。

常见报错

  • 现象：ModuleNotFoundError: eval / backend → 你不在仓库根（mini-openclaw/）跑，或没 conda activate openclaw。用 python -m eval.xxx 从根目录以模块方式运行，别 cd eval 再 python metrics.py。
  • 现象：步骤 3 报"缺少 DEEPSEEK_API_KEY" → export DEEPSEEK_API_KEY=sk-... 后重跑；只有步骤 3（judge）真的调 API，其余步骤纯本地、无需 key。
  • 现象：judge 的分数解析成 None → 模型没按"单独一行【分数: X】"输出；把 RUBRIC 里的格式约束写得更死（强调"最后一行只写【分数: X】"），或放宽 re 正则。judge 输出不稳本身就是 §9.3 说的"一致性差"，正好体会。
  • 现象：success_rate 恒为 1.0 或 0.0 → 检查 check 函数是否真在读 traj["steps"] / traj["final"]；_check_domain 若还是 raise NotImplementedError 会让整条记录判失败。
  • 现象：json_valid_rate 结果不符合预期 → 确认坏 JSON 那条样本的 raw 里确实有 <tool_call> 开头（否则不计入分母）；json.loads 失败时应跳过而非抛出。
  • 现象：eval/trace_sample.jsonl 是空的 → Tracer.__init__ 会先清空文件，确认之后确实调了 log_step；replay 读的路径要和写入路径一致。
  • 现象：想直接接真 agent → 现在还不能——主循环是 D4 的产出。今天所有验证都在样本轨迹上完成，这是刻意的：先把度量工具打磨好，D4 一有真轨迹就能立刻量它。
