Day 2 下午 · 连通后端 + 首个工具 schema 往返（M2）

一句话目标：打通 mini-OpenClaw 的"大脑"——DeepSeek API，并让它认识"工具"这个概念。 本次产出：① 用 backend/client.py 调通一次 DeepSeek 对话 ② 定义第一个工具的 JSON schema、传入 tools 并拿回响应 ③ 在 agent/prompts.py 写出系统提示词雏形。 涉及的 starter 文件：backend/client.py（读+调用）、agent/prompts.py（SYSTEM_PROMPT，其后有 # TODO[Day4] 标记留待后续）。本日为本日构建目标（不单独计分，计入最终展示）。

开始之前

  • 进环境： 
    cd mini-openclaw && conda activate openclaw
export DEEPSEEK_API_KEY="sk-..."     # Day1 已配则跳过
  • Day2 主要是读 backend/client.py、调用它，并起草 agent/prompts.py 的 SYSTEM_PROMPT（没有 TODO[Day2] 标记，SYSTEM_PROMPT 就写在 # TODO[Day4] 之前那段）。定位它： 
    grep -rn "SYSTEM_PROMPT" agent/prompts.py
grep -rn "def chat" backend/client.py
  • 今天不追求完整主循环（那是 Day4），目标是把"模型 + 工具 schema"这条链路先打通、看明白。

步骤 1 · 连通 DeepSeek 后端（文件：backend/client.py）

  1. 先读一遍 backend/client.py：看清 DeepSeekBackend.chat(messages, tools=None, temperature=0.0) 怎么把 messages/tools 组织成请求、又怎么把返回归一化成 {"role": "assistant", "content": str, "tool_calls": [...]}。
  2. 在仓库根目录调一次，验证 API 通： 
    python -c "from backend.client import DeepSeekBackend; print(DeepSeekBackend().chat([{'role':'user','content':'用一句话自我介绍'}]))"
  3. ✅ 验证：打印出一个字典，content 字段里有模型的中文回复、tool_calls 为 []。若报错，先查 DEEPSEEK_API_KEY、DEEPSEEK_BASE_URL、网络。

步骤 2 · 定义第一个工具的 schema 并往返（新建练习脚本 demo_m2.py）

模型不会"真的调用函数"，它只是看到 tools 描述后，输出一段工具调用文本。这一步把"工具描述"喂给它、看它怎么回。

  1. 在仓库根新建 demo_m2.py，用 OpenAI tools 格式定义一个最简单的工具 get_time： 
    from backend.client import DeepSeekBackend

tools = [{
    "type": "function",
    "function": {
        "name": "get_time",
        "description": "返回当前时间。用户询问现在几点、当前时间时使用。",
        "parameters": {"type": "object", "properties": {}, "required": []},
    },
}]

backend = DeepSeekBackend()
resp = backend.chat(
    [{"role": "user", "content": "现在几点？"}],
    tools=tools,
)
print(resp)
  2. 运行并观察返回： 
    python demo_m2.py
  3. 看清返回结构：tool_calls 是不是非空？其中每一项的 name（应为 get_time）和 arguments（一个 dict）长什么样？
  4. ✅ 验证：resp["tool_calls"] 里出现一项 name == "get_time" 的调用。若模型这次直接用文字答了、没触发工具，也不算错（链路已通）——把问题写得更"非工具不可"或再跑一次即可。 关键理解：tools 字段最终会被拼进模型看到的 prompt（由服务端按对话模板完成），今天先建立"schema 进去 → tool_calls 出来"的直觉。

步骤 3 · 起草系统提示词雏形（文件：agent/prompts.py）

  1. 打开 agent/prompts.py，找到 SYSTEM_PROMPT（在 # TODO[Day4] 之前那段，已有一个最小起点）。
  2. 把它改写成你 agent 的初版，说清三件事： - 角色：一个运行在用户工作目录下的命令行智能体。 - 能力声明：能调用工具读写文件、执行 shell、搜索代码、抓取网页等。 - 行为准则：一次只做一小步、依赖工具结果再决策、失败时读报错再修、完成后简洁作答。 - 保留其后的 # TODO[Day4]（含后续上下文管理）注释不动（后续继续打磨）。
  3. ✅ 验证：导入不报错、能打印出你写的提示词—— 
    python -c "from agent.prompts import SYSTEM_PROMPT; print(SYSTEM_PROMPT)"

步骤 4 · 串起来跑一遍并提交（文件：demo_m2.py）

  1. 在 demo_m2.py 末尾把三件事串起来验证一次：连通 ✓、tools 往返 ✓、系统提示词就位 ✓。可把 SYSTEM_PROMPT 作为一条 {"role": "system", ...} 消息放到 messages 最前面再调一次： 
    from agent.prompts import SYSTEM_PROMPT

resp2 = backend.chat(
    [{"role": "system", "content": SYSTEM_PROMPT},
     {"role": "user", "content": "现在几点？"}],
    tools=tools,
)
print(resp2)
  2. 跑通后提交： 
    python demo_m2.py
git add -A && git commit -m "day2: M2 backend + first tool schema"
  3. ✅ 验证：git log --oneline 能看到这次提交，demo_m2.py 一次运行内三件事都不报错。

完成检查（本日构建目标，不单独计分，计入最终展示）

  • ☐ 步骤 1：一次不带 tools 的 chat 调用能返回模型回复。
  • ☐ 步骤 2：一次带 tools 的 chat 调用能拿回响应，你能指出返回里的 tool_calls（或说明为何本次没触发）。
  • ☐ 步骤 3：agent/prompts.py 的 SYSTEM_PROMPT 已写成你 agent 的初版，说清了"我是谁、能做什么、怎么做"。
  • ☐ 步骤 4：demo_m2.py 串通三件事并已提交。
  • ☐ 你能用一句话解释：tools 字段是怎么影响模型输出的？（答到"它作为文本进入 prompt，模型据此决定是否输出工具调用文本"即可）相关评分见 ../00-课程总览/考核与评分总表.md §2。

常见报错

  • 现象：API 报 401 / 额度不足 → key 错或额度用尽；换 key、查账户额度。
  • 现象：不返回 tool_calls，模型直接答了 → 把问题写得更"非工具不可"（如"现在几点"），或确认 tools／tool_choice 传对；模型不触发工具也不算错，重点是链路通。
  • 现象：ModuleNotFoundError → 你不在仓库根目录，或没 conda activate openclaw。
  • 现象：连接超时／网络错误 → 检查 DEEPSEEK_BASE_URL 与网络代理设置。
