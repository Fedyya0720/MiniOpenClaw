Day 1 下午 · 环境搭建与系统观察

一句话目标：把开发环境搭好、让 mini-OpenClaw 骨架自检通过，并观察 Claude Code 内部是怎么干活的。 本次产出：① Claude Code 观察笔记（自用，不提交）② 分组与领域登记 ③ 能跑通 python -m agent.cli --selfcheck 的 mini-OpenClaw 仓库。 涉及的 starter 文件：agent/cli.py（--selfcheck）、backend/client.py（DeepSeek 配置）。本日为本日构建目标（不单独计分，计入最终展示）。

开始之前

  • 你需要：一台能联网的电脑、已装好 conda（或 miniconda）、git、Node.js ≥ 18（步骤 1 装 Claude Code 用）。
  • 一把 DeepSeek API key（sk- 开头）：它既驱动步骤 1 的 Claude Code，也驱动你后面每天要搭的 mini-OpenClaw。没有就先找课程领一把。
  • 动手前先过一遍 Claude-Code-使用技巧.md，重点看：基础工作流、/plan /diff /export 命令、"导出会话轨迹"一节（步骤 2 会用）。
  • 各天增量依赖、API 退路见文末 附录。

步骤 1 · 准备 Claude Code（用你的 DeepSeek key 驱动）

你不需要 Anthropic 账号——Claude Code（CC）直接用你已有的 DeepSeek key 驱动。原理：DeepSeek 提供 Anthropic 兼容接口（https://api.deepseek.com/anthropic），CC 只认几个 ANTHROPIC_* 环境变量，指过去即可。

  1. 安装 CC（需 Node.js ≥ 18）并确认版本： 
    npm install -g @anthropic-ai/claude-code
claude --version
  2. 让 CC 走 DeepSeek——把下面几行写进你的 shell 配置（~/.zshrc / ~/.bashrc），然后 source 生效。其中 DEEPSEEK_API_KEY 就是步骤 5 要配的那把 key（可现在先配好）： 
    export DEEPSEEK_API_KEY="sk-你的key"
export ANTHROPIC_BASE_URL="https://api.deepseek.com/anthropic"
export ANTHROPIC_AUTH_TOKEN="$DEEPSEEK_API_KEY"
export ANTHROPIC_MODEL="deepseek-v4-pro"           # 主模型：更强
export ANTHROPIC_SMALL_FAST_MODEL="deepseek-v4-flash"  # 轻量快模型
  3. 新建一个空目录，在里面运行 claude 启动；可先 /init 让它读一遍目录。
  4. ✅ 验证：CC 进入交互界面后，发一句「你好，用一句话介绍你自己」，能收到 DeepSeek 的回复即说明整条链路（CC → DeepSeek）通了。若装不上或接口不通，用现场提供的轨迹样本完成步骤 2 的观察（见 Day1观察活动.md）。

步骤 2 · 用 Claude Code 完成一个任务并观察它

观察目标不是"把活干完"，而是看清它内部每一步在做什么——为上午那张四层架构图找真实证据。这一步的所有笔记都不提交。

  1. 从下面任选一个任务（或按现场指定），交给 Claude Code 完成： - 代码修复：给一个有 bug 的小脚本（如埋了 KeyError／缩进错），让它定位并修复到能运行。 - 日志分析：给一份报错日志，让它找出异常原因并给出修复建议。 - 代码库导读：给一个陌生小 repo，让它总结结构和每个模块作用。
  2. 全程盯住它的每一步，对照上午的四层架构（LLM / 主循环 / 工具系统 / 权限层）记录： - 它调用了哪些工具？每个工具的输入输出长什么样？ - 它在哪一步做了决策（选 A 工具而不是 B）？依据是什么？ - 哪一步停下来征求许可？（权限层证据） - 有没有失败后自我修复的片段？（错误恢复证据） - 它一共和模型交互了几轮？ 3.（可选）留一份完整轨迹方便回看：在 Claude Code 内运行 /export trace.txt，导出为易读纯文本（含消息／工具调用／输出）。详见 Claude-Code-使用技巧.md §7。
  3. ✅ 验证：你的笔记里至少写下 ≥3 项关键设计决策。活动指引见 Day1观察活动.md。这份笔记自用，不提交。

步骤 3 · 分组与领域登记

  1. 2–3 人自由组队。
  2. 选定一个领域方向（全程不变，影响 Day3 评测任务集、Day5 Skill、Demo Day）：
方向 典型任务 Day5 Skill 例子 Demo Day 现场任务示例
科研辅助 读论文、整理文献、跑实验脚本 "论文速读报告" 给一篇论文 PDF/文本，产出结构化要点
数据分析 CSV/日志统计、出图、写报告 "CSV 快速报告"（starter 已带示例） 给一个 CSV，产出统计报告
日志排障 定位异常栈、grep 报错、给建议 "异常栈定位" 给一段报错日志，定位原因并建议修复
代码库导读 陌生 repo 摸索、模块说明、找 TODO "陌生仓库导读" 给一个小 repo，生成模块说明

步骤 4 · 拿到 starter 骨架

全组共用一个 git 仓库。今天只需让骨架自检通过，后面逐天往里填。

  1. 获取 starter（现场发的压缩包或 git 地址），复制成你们组的仓库并初始化： 
    cp -r <starter路径> mini-openclaw      # 或 git clone <地址> mini-openclaw
cd mini-openclaw
git init && git add -A && git commit -m "day1: init skeleton"
  2. 通读 README.md 了解骨架结构： 
    backend/  agent/  tools/  mcp/  skills/  eval/  prompt/  requirements.txt
  3. ✅ 验证：ls 能看到上述目录，且当前工作目录在 mini-openclaw 仓库根。

步骤 5 · 建环境、配 API key

  1. 创建并激活 conda 环境，装依赖（agent 依赖很轻：httpx/pydantic/markdownify）： 
    conda create -n openclaw python=3.11 -y && conda activate openclaw
pip install -r requirements.txt
    再装系统工具 ripgrep（Day4 的 grep 工具会用）：brew install ripgrep（macOS）/ apt install ripgrep（Linux）。其余各天的增量依赖见文末附录，按需再装。
  2. 配置 DeepSeek API key（写进你的 shell rc，别提交进 git）。若步骤 1 已配过就跳过： 
    export DEEPSEEK_API_KEY="sk-你的key"
    相关环境变量（backend/client.py 读取）：DEEPSEEK_API_KEY、DEEPSEEK_BASE_URL（默认 https://api.deepseek.com）、DEEPSEEK_MODEL（默认 deepseek-v4-flash，轻量快、够用；更强可设 deepseek-v4-pro）。
  3. ✅ 验证（可选，需要 key）：确认 key 能成功调一次—— 
    python -c "from backend.client import DeepSeekBackend; print(DeepSeekBackend().chat([{'role':'user','content':'说你好'}]))"
    打印出一个含 content 的字典即说明 API 通了。没 key 也不影响下一步。

步骤 6 · 跑通骨架自检

  1. 在仓库根目录运行自检（agent/cli.py 的 --selfcheck 分支，不需要 key）： 
    python -m agent.cli --selfcheck
  2. ✅ 验证：看到下面这样的输出，最后一行为 通过 ✅：

[ok] FakeBackend 可用（未配 DEEPSEEK_API_KEY 时的离线占位后端）
[ok] 主循环模块可导入（Day4 实现 run 逻辑）

步骤 8 · 部署上线（把 mini-OpenClaw 跑到平台上）

课程提供一个部署平台：上传 starter.zip 就能把你们组的 agent 跑成一个网页终端，别人打开链接即可用。今天先用 --selfcheck 跑通整条部署流程，Demo Day 直接复用同一套。

  1. 登录：打开 http://10.130.130.9:8000，用你们的组号账号登录（如 g01）。
  2. 部署（3 步）： - 上传 starter.zip（你们组当前的仓库打包；无需自己写 Dockerfile，平台自动补）。 - 填环境变量：DEEPSEEK_API_KEY=sk-...（必填）；可选 DEEPSEEK_MODEL / DEEPSEEK_BASE_URL。 - 点 「构建并部署」，等状态变绿（构建 + 启动完成）。
  3. 展示 / 自测：把你们的地址 http://10.130.130.9:80NN（NN 为你们组号）发给观众，打开即是一个网页终端，在里面运行： 
    python -m agent.cli --selfcheck
python -m agent.cli "创建 hello.py 并运行"

    说明：--selfcheck 今天就能通过；"创建 hello.py 并运行" 依赖 read/write/bash 等工具，要到 Day4 起才逐步可用。今天跑通 --selfcheck 即达标，第二条留作后续里程碑的现场验收。

  4. ✅ 验证：平台状态为绿色，网页终端里 python -m agent.cli --selfcheck 输出 == 自检 通过 ✅ ==。

常见报错

  • 现象：claude 命令找不到 → Node.js 未装或版本 < 18，或全局 npm bin 不在 PATH；node -v 确认版本，必要时把 npm bin -g 的路径加进 PATH。
  • 现象：CC 报鉴权/401 或连接错误 → ANTHROPIC_* 没配全或没 source；确认 ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic、ANTHROPIC_AUTH_TOKEN 里是有效的 DeepSeek key、echo $ANTHROPIC_MODEL 为 deepseek-v4-pro。
  • 现象：Claude Code 装不上／接口连不上 → 别卡在环境上，用现场提供的轨迹样本完成步骤 2 的观察（见 Day1观察活动.md）。
  • 现象：python -m agent.cli 报 ModuleNotFoundError → 你不在 mini-openclaw 仓库根目录，或没 conda activate openclaw；cd 回仓库根、重新激活环境。
  • 现象：pip install -r requirements.txt 失败 → 确认已激活 openclaw 环境、网络可用；单独重试报错的那个包。
  • 现象：步骤 5 的 key 自测报错 → 检查 DEEPSEEK_API_KEY 是否已 export、值是否正确、网络能否访问 DEEPSEEK_BASE_URL。没 key 时跳过此项即可，--selfcheck 仍会通过。
  • 现象：找不到导出的轨迹文件 → ls ~/.claude/projects/ 看有哪些项目目录，找最新修改的 .jsonl；能用 /export 就优先用它。
  • 现象：平台部署状态一直不变绿 / 构建失败 → 多半是 starter.zip 打包时把仓库根目录多套了一层，或漏了 requirements.txt；确认解压后根目录直接是 agent/ backend/ tools/ requirements.txt。运行期报鉴权错则是环境变量 DEEPSEEK_API_KEY 没在平台里填。
