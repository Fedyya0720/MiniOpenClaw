考核与评分总表

总分 100%，构成极简： 最终展示 95%（Demo Day 现场演示 + 讲解 + 答辩，附源码/消融/文档） ＋ 课堂参与 5%。 课程按天推进构建 mini claude code（见各 dayNN/lab-guide.md 的「本日构建目标」）， 但不按天单独计分；十天的全部工作，最终统一在 Demo Day 的展示中评定。

1. 权重构成

考核项目 权重 形式
最终展示 95% Day10 Demo Day：现场演示 + 讲解 + 答辩；提交源码 + 消融报告 + 技术文档
课堂参与 5% 全程：出勤 + 每日实验课投入 + 课堂互动
合计 100%
  • 每日构建目标不再单独计分，仅作为逐日推进的路线图与自检清单（见各天 lab-guide）。成绩只体现在最终展示与课堂参与中。
  • 最终展示以小组为单位；课堂参与为个人分。
  • 建议按里程碑打 tag：v1(Day4) / v3(Day5) / final(Day6 安全层)，其后 Day7–9 继续在 final 上叠加记忆 / 规划 / 可观测，便于展示时回溯。

2. 最终展示评分细则（95 分）

Day10 Demo Day，每组约 15–20 分钟：现场演示（含当场指定的随机任务）+ 实现讲解 + 答辩，并提交源代码 + 消融实验报告 + 技术文档。 下面 8 类打分点合计 95 分，直接计入总评。组内每位成员都应能参与讲解与答辩。

A. 系统完整性与可运行性（18 分）

打分点 分 满分要求
A1 一键启动可运行 6 python -m agent.cli "任务" 能直接运行，环境/依赖说明清楚
A2 架构完整 8 覆盖 backend（DeepSeek 客户端）/ 主循环 / 工具系统 / MCP / Skills / 安全层 / 记忆(D7) / 规划(D8) / 可观测(D9)，各层齐全
A3 代码结构与可复现 4 结构清晰、模块分层合理；README + 复现步骤可照做

B. 现场任务完成度（18 分）

打分点 分 满分要求
B1 随机任务完成 12 当场指定、未提前透露的任务能自主完成且结果正确
B2 多工具协同 6 合理编排 glob/grep/read/edit/write/bash 等，而非硬靠单一工具

C. 主循环、规划与工具调用正确性（14 分）

打分点 分 满分要求
C1 主循环与长任务规划 8 多轮调用、tool result 正确回填、正确终止、无死循环；长任务能列 todo、逐条推进、不漏不重（D8）
C2 工具集正确 6 read/write/bash/edit/grep/glob 按预期工作；edit 匹配稳、grep/glob 互补

D. 扩展能力：MCP + Skills（12 分）

打分点 分 满分要求
D1 MCP 接入 6 客户端能连真实 server（echo/filesystem 等），工具透明并入主循环
D2 Skills 6 加载器可用；领域定制 Skill 能被正确召回并按流程执行

多模态（图像输入，Day 5 下半场）不单列打分点；若实现——agent 能接收并推理图像（如看截图排 UI bug）——计入系统完整性（A）与现场任务（B）的加分表现。

E. 上下文 / 记忆与鲁棒性（10 分）

打分点 分 满分要求
E1 上下文与记忆 5 compaction / tool result 截断生效、长任务不撑爆；跨会话记忆生效（D7：新会话仍记得项目约定）
E2 错误恢复 5 工具失败时把报错作为 observation 喂回、自我修复，不崩溃；单步失败不拖垮全局（D8）

F. 安全机制（8 分）

打分点 分 满分要求
F1 权限分层 4 按操作破坏性分级：只读放行 / 写限目录或确认 / 执行·外传需确认；危险命令能拦截
F2 注入防护 + 红队 4 对不可信内容（网页/文件）做隔离，并现场演示一次注入被挡住的红队用例

G. 技术理解与现场答辩（9 分）

打分点 分 满分要求
G1 实现与设计讲解 5 讲清每部分如何实现、为何这样设计（关键取舍：edit 策略 / compaction / 权限分层 / 记忆 / 规划等）；能用 trace / 成本（D9）佐证
G2 答辩 4 准确回答现场提问；组内每位成员都能回答自己负责部分的机理问题

H. 消融实验与技术文档（6 分）

打分点 分 满分要求
H1 消融实验 4 ≥1 组消融（有/无 compaction、有/无 todo 规划、有/无记忆、工具描述好坏、有/无错误恢复），有数据有结论——用数据证明某设计确实有用
H2 技术文档 2 架构说明 + 关键设计决策 + 各模块 README；含安全小节与可观测 / 成本小节（D9）

合计 95 分。

3. 评分分档（参考）

  • 90–95：系统完整可运行，随机任务漂亮完成，MCP/Skills/安全层齐备并能演示；讲解与答辩清晰，消融有数据支撑。
  • 75–89：主体功能可用，随机任务基本完成，但某一两块（常见：安全层、消融、或扩展能力）较弱或演示不充分。
  • 60–74：能演示基本的主循环 + 工具，但扩展能力/鲁棒性欠缺，答辩浅。
  • <60：系统无法自主完成任务，或关键模块缺失/未跑通。

4. Demo Day 现场组织

  • 每组 15–20 分钟：约 5 分钟讲架构与设计亮点 → 8–10 分钟现场任务（含当场指定的随机任务）→ 答辩。
  • 随机任务：评委当场给一个未提前透露、偏该组领域方向的任务（题池见 day10-DemoDay验收/demo-day指南.md）。
  • 鼓励现场制造一次失败（改个路径让某工具失败），观察错误恢复——恢复过程往往比一次成功更能体现功力。
  • 组织与题池细节见 day10-DemoDay验收/demo-day指南.md。

5. 课堂参与（5%）

出勤 + 每日实验课投入 + 课堂互动 / 每日进度检查时的表现。

6. 学术诚信

  • 允许用 AI 辅助写代码（这本就是一门关于 agent 的课），但消融报告与技术文档须体现自己的理解；Demo Day 的现场随机任务与答辩考的是真功夫。
  • 答辩为现场，组内每位成员都应能回答自己负责部分的机理问题——避免只有一人真正掌握。
