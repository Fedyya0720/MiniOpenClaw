Day 9 下午 · 可观测性层：Tracer + 回放 + 成本核算 + trace 调试 + 优化

一句话目标：给 agent 装上仪表盘——把每一步结构化成可回放的 trace，算清 token 账，用 trace 定位一个真实 bug，并做至少一项性能优化。 本次产出：一个 Tracer（把 LLM/工具调用记成 span）+ trace 回放 + token/成本核算（每轮+累计+最贵一步）+ 用 trace 定位并修一个 bug + 一项优化（前缀缓存友好排序 / 并行工具调用）。 涉及的 starter 文件：新建 agent/tracer.py；接线在 agent/loop.py（包住 chat 与工具执行）。（本课新增内容，starter 暂无 TODO[Day9] 桩，按下面步骤新建即可。）

开始之前

  • 所有命令都在 starter/ 目录下运行：cd starter。
  • 承接前几天：agent 已有主循环、工具、MCP、Skills、安全、记忆、规划。
  • 先想清楚边界（讲义 §2）：今天是给开发者看过程（trace/度量/调试），不是评估(D3)、不是上下文管理(D4)、不是安全审计(D6)。

步骤 1 · Tracer：把每次调用记成 span（新建 agent/tracer.py）

  1. 一次运行 = 一串 span（讲义 §4）。用一个统一包裹自动采集，不必处处埋点： 
    import time

class Tracer:
    def __init__(self):
        self.spans = []

    def span(self, kind, name, fn, **meta):
        t0 = time.time(); ok, out = True, None
        try:
            out = fn(); return out
        except Exception as e:
            ok, out = False, repr(e); raise
        finally:
            self.spans.append({"kind": kind, "name": name, "ok": ok,
                               "ms": round((time.time() - t0) * 1000),
                               "out": str(out)[:500], **meta})
  2. 接线（agent/loop.py）：把每次 LLM 调用与工具执行包起来： 
    resp = tracer.span("llm", "decide", lambda: backend.chat(messages),
                   tokens=None)                       # tokens 在步骤 3 补
result = tracer.span("tool", call.name, lambda: registry.run(call.name, call.args))
  3. ✅ 验证：跑一个多步任务后 print(len(tracer.spans)) 应 > 0，且交替出现 llm / tool。

步骤 2 · 回放（replay）：把一次运行渲染出来（agent/tracer.py）

  1. 有了 spans 就能回放整次运行——调试的起点： 
    def replay(tracer):
    for i, s in enumerate(tracer.spans, 1):
        tok = ("  %stok" % s.get("tokens")) if s.get("tokens") else ""
        flag = "" if s["ok"] else "  ✗"
        print("#%d %-4s %-10s %5dms%s  → %s%s"
              % (i, s["kind"], s["name"], s["ms"], tok, s["out"][:40], flag))
  2. ✅ 验证：replay(tracer) 能一眼看清"它到底怎么走的"——每步的类型、名称、耗时、输出。

步骤 3 · token 与成本核算（agent/tracer.py）

  1. 从模型返回的 usage 取 prompt_tokens / completion_tokens，记进 LLM span，并累加（讲义 §6）： 
    resp = backend.chat(messages)
usage = resp.get("usage", {})
# 把 usage 塞进最近一个 llm span（或在 span() 调用时传 tokens=usage["total_tokens"]）

def cost_report(tracer, price_per_1k=0.001):
    llm = [s for s in tracer.spans if s["kind"] == "llm" and s.get("tokens")]
    total = sum(s["tokens"] for s in llm)
    priciest = max(llm, key=lambda s: s["tokens"], default=None)
    print("总 token:", total, " 估算成本: $%.4f" % (total/1000*price_per_1k))
    if priciest:
        print("最贵一步: span %s，%d tok" % (priciest["name"], priciest["tokens"]))
  2. 观察二次膨胀（讲义 §6.2）：打印每轮的 prompt_tokens，你会看到它随轮数越来越大（每轮重发全历史）。
  3. ✅ 验证：cost_report 能报出总 token 与最贵的一步；每轮 prompt_tokens 单调上升。

步骤 4 · 用 trace 调试一个真实 bug（agent/loop.py）

  1. 制造 / 复现一个经典 bug（讲义 §5.3）：tool result 没回填——注释掉主循环里"把工具输出加回 messages"的那一步，跑一个任务。
  2. 回放 trace，你会看到：某个 llm span 之后重复调用同一个工具（因为它没看到上一步的结果）。据此定位。
  3. 修复：把工具输出作为 observation 回填进下一轮 messages，重跑，确认重复消失。
  4. ✅ 验证：修复前 trace 里有重复的 (工具+参数) span；修复后没有。这就是"把猜变成查"。

步骤 5 · 至少一项优化（agent/prompts.py / agent/loop.py）

任选其一（或都做）：

  • 前缀缓存友好排序（讲义 §7.1）：把稳定内容（system prompt、skill catalog、项目记忆）放在上下文最前面，每轮变化的内容（历史、观察）放后面。别把时间戳等每轮变化的东西塞在最前——否则前缀一变、整段缓存失效。 
    # 验证：连续两轮请求的开头一大段应完全相同（可命中前缀缓存）
  • 并行工具调用（讲义 §7.2）：当模型一轮里请求多个无依赖工具调用时，用线程池并发执行、汇总回填： 
    from concurrent.futures import ThreadPoolExecutor
with ThreadPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(lambda c: registry.run(c.name, c.args), calls))
  • ✅ 验证：前缀缓存——两轮请求的公共前缀显著变长；并行——多个 read 的总耗时接近单个而非之和。

  • ☐ token/成本核算：报出总 token 与最贵一步；能观察到每轮 prompt_tokens 二次膨胀。
  • ☐ 用 trace 定位并修复一个真实 bug（如 tool result 没回填导致的重复调用）。

常见报错

  • 现象：span 没采集到 → fn() 抛异常时用了 except 吞掉 → 用 finally 记录、异常照常 raise（见步骤 1）。
  • 现象：token 一直是 None → 没从 resp["usage"] 取 → 每次 chat 后把 usage 记进对应 llm span。
  • 现象：回放太长刷屏 → span 的 out 没截断 → 存进 span 时就 [:500]，回放再 [:40]。
  • 现象：并行后结果错乱 → 有依赖的调用被并行了 → 只并行无依赖的；有依赖的仍串行。
  • 现象：优化了但缓存不命中 → 前缀里混了每轮变化的内容 → 把稳定内容严格放最前，变化内容放后。
  • 现象：分不清该记 trace 还是审计日志 → 目的不同（调试 vs 问责），但可共用底层记录，加不同视图即可（讲义 §2）。
