# PACS TUI 消融测试简报

日期：2026-07-13  
环境：macOS，`deepseek-v4-flash`，TUI `--auto-approve --workdir`  
重复：每个项目、每种策略各 2 次，共 20 次独立 TUI 运行

## 实验设置

每次运行使用 `Desktop/test` 项目的全新 `/tmp` 副本和独立 HOME，不复用项目内
PACS 缓存、虚拟环境、lock 或 Agent trace。计时区间为“提交自然语言请求”到“TUI
再次出现输入提示符”的端到端时间，包含模型调用、工具调用、下载、安装、验证和最终答复。

- 默认组：`请为当前 Python 项目自动配置一个可运行的环境。直接完成，不要只给步骤。`
- 传统组：明确要求不调用 `pacs_build` 或其他 PACS 工具，使用传统方式创建 venv 并安装依赖。

同一项目同一策略的两次重复并发启动，以缩短测试时间；因此外部 API/PyPI 网络争用可能影响
绝对时间。传统组额外以 `pip check` 和关键包存在性检查审计真实结果，不能只看 Agent 最终文本。

## 结果

| 项目 | 默认组两次 / s | 默认均值 / s | 默认结果 | 传统组两次 / s | 传统均值 / s | 传统结果 | 传统÷默认 |
|---|---:|---:|---|---:|---:|---|---:|
| sampleproject | 31.22 / 29.46 | 30.34 | PACS 2/2 | 62.52 / 66.55 | 64.54 | 2/2 | 2.13× |
| uv-fastapi-example | 81.23 / 82.00 | 81.61 | PACS 2/2 | 146.59 / 269.78 | 208.19 | 1/2 | 2.55× |
| requests | 28.62 / 29.12 | 28.87 | PACS 2/2 | 75.52 / 88.53 | 82.03 | 2/2 | 2.84× |
| flask | 29.62 / 28.22 | 28.92 | PACS 2/2 | 31.83 / 225.67 | 128.75 | 1/2 | 4.45× |
| poetry | 579.86 / 616.31 | 598.09 | PACS 0/2；Agent 降级后 2/2 | 125.83 / 187.36 | 156.60 | 2/2 | 0.26× |

汇总：

- 20 次运行的中位数：默认组 **30.42 s**，传统组 **107.18 s**，默认组约快 **3.52×**。
- 排除 Poetry：默认组均值 **42.44 s**，传统组 **120.87 s**，默认组约快 **2.85×**。
- 包含 Poetry 的总体均值：默认组 **153.57 s**，传统组 **128.02 s**；该均值被 Poetry 的 PACS 失败和后续 Agent 降级显著拉高。
- PACS 本身成功率 **8/10**；默认 TUI 最终成功率 **10/10**，但 Poetry 两次都是 PACS 失败后改走低层/传统工具。
- 传统组严格验证成功率 **8/10**：一次 FastAPI 环境 `pip check` 失败，一次 Flask 只创建了空 venv、未安装项目依赖。

## 观察

1. 对四个小中型项目，PACS 的速度和重复稳定性都明显更好。正常路径通常只需 1 次
   `pacs_build`（sampleproject 有一次额外报告读取），传统组需要 12～25 次 glob/read/bash。
2. 传统 Agent 路径波动很大。Flask 两次相差约 194 秒；失败路径中出现了反复重建 venv、
   切换镜像、`--no-deps` 手工补包等行为，既慢又容易得到表面成功但实际损坏的环境。
3. Poetry 是当前 PACS 的明显反例。一次先遇到 PyPI JSON SSL EOF，后续候选又在
   `rapidfuzz`/复杂传递依赖、git 形式 `poetry-core` 和安装阶段失败；Agent 随后进行了约
   21 次额外工具调用才完成环境，导致端到端接近 10 分钟。
4. 当前 PACS 的 `success` 代表依赖安装和 `pip check` 通过；它尚未默认 editable 安装目标
   项目或运行项目测试，因此本实验衡量的是“依赖环境装配”，不是完整应用测试时间。

## 结论与建议

当前 Demo 已证明 PACS 对常规项目能显著减少模型回合、提升速度并降低方差，但还不能根据
5 个项目的总体均值宣称全面更快。下一轮应优先优化 Poetry 暴露的问题：并行且可重试的 PyPI
元数据获取、lockfile/直接引用处理，以及复杂项目失败后的受控降级。PACS 业务失败后也应由
Agent 直接汇报，或只进入一个明确、有限的 fallback，而不是恢复成开放式手工探索。

`scripts/benchmark_tui.exp`。
