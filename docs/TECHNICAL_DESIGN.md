# MiniOpenClaw 技术设计说明

## 1. 项目目标

MiniOpenClaw 是一个运行在命令行中的通用智能体。用户用自然语言描述任务，大模型判断下一步是直接回答还是调用工具；程序负责检查权限、执行工具、保存结果，并把执行结果送回模型，形成“判断—行动—观察”的闭环。

本项目在通用智能体之外实现了 PACS（Parallel Adaptive Compatibility Search，并行自适应兼容性搜索）子系统，重点处理 Python 项目的环境配置问题。它读取项目中的依赖声明，生成有限数量的版本候选，在隔离的虚拟环境中并行验证，依据真实失败日志记录不兼容组合，最后输出一个通过校验的环境和锁定文件。

我们针对该“能自动配置环境的Agent”的设计的核心理念是————自然语言理解适合交给大模型，但依赖解析、候选求解、安装、验证和证据记录应尽量由确定性代码完成。这样既保留了 Agent 的交互能力，也避免让环境配置完全依赖模型临场猜测，减少模型的幻觉率，提高模型的工作效率。

项目的目标不是实现一个生产级容器平台或完整的 Python 包管理器，而是在结构清晰、行为可观察的前提下，验证通用 Agent 与确定性依赖搜索结合的可行性。

## 2. 总体架构

系统分为交互层、Agent 核心层、能力扩展层和 PACS 执行层。

```text
用户
 │
 ├─ 单次 CLI / 交互式 TUI
 │
 ▼
Backend（DeepSeek 或 FakeBackend）
 │  返回文本、工具调用和 token usage
 ▼
ReAct 执行策略
 ├─ 上下文预算与压缩
 ├─ 权限判断
 ├─ 工具调度
 ├─ 长输出文件化保存
 ├─ 工具trace记录
 └─ span、耗时与 token 记录
 │
 ▼
ToolRegistry
 ├─ 文件、命令、检索、网页和任务列表
 ├─ 项目记忆与 Skill
 ├─ MCP 外部工具
 └─ PACS 高层入口及底层环境工具
```

CLI 和 TUI 只负责输入输出形式。两者最终都进入 `run_react_turns`，权限判断、工具调用、上下文处理和 trace 行为保持一致。

PACS 的内部流程如下：

```text
解析项目依赖
   │
   ▼
构建版本目录并限制搜索窗口
   │
   ▼
候选求解与可解释评分
   │
   ▼
并发 preflight
   ├─ 失败：解析日志，记录已观察约束，重新求解
   └─ 成功：进入并行环境安装
                    │
                    ▼
          安装项目并运行 pip check
                    │
          可选：导入指定模块验证
                    │
         ┌──────────┴──────────┐
         │                     │
       成功                  失败
         │                     │
   保留环境和 lock       学习失败并继续搜索
   清理其他环境
```

Agent 可以通过一个 `pacs_build` 工具触发完整流程。搜索循环由 `PACSBuilder` 持有，而不是让模型逐步安排每一次 `pip install`。二者的职责有一个清晰的边界：模型适合识别用户想配置哪个项目、是否需要指定 Python 版本；候选数量、并发上限、重试次数和成功判据则应由程序控制。

## 3. Agent 核心流程

### 3.1 后端抽象

`DeepSeekBackend` 对 OpenAI 兼容的 Chat Completions 接口进行封装，对外提供 `chat(messages, tools)` 和流式 `chat_stream(messages, tools)`。后端把不同响应格式归一化为正文、工具调用和 usage，Agent 循环不需要理解 HTTP 与 SSE 细节。

网络超时、连接中断、HTTP 429 和部分 5xx 状态会进行有限次数的指数退避重试。参数错误和鉴权失败属于客户端问题，不进行无意义重试。没有 API Key 时可使用 `FakeBackend` 完成离线自检，但它只能验证程序链路，不能代表真实模型任务效果。

### 3.2 ReAct 循环

每一轮先检查上下文预算，再调用模型。模型没有返回工具调用时，正文就是最终结果；否则程序依次执行本轮工具，将结果作为 `role="tool"` 的 observation 加入消息历史，再进入下一轮。

主循环设置默认 20 轮的硬上限。这会虽然会牺牲少数超长任务的完成空间，但可以防止模型在重复失败时无限消耗时间和 token。工具不存在、权限拒绝和工具异常都被转换为可读 observation，而非让进程直接退出，使得模型有机会修改参数或改用其他方案。同时，trace 中仍会记录失败状态，方便审查校验。

同一轮中的普通工具目前按顺序执行。PACS 的安装并发由子系统内部控制，因为它可以统一限制并发度、清理环境和保存日志；如果在通用 ReAct 层盲目并行所有工具，存在写入冲突和操作顺序被破坏的风险。

### 3.3 工具系统

工具由名称、说明、JSON Schema 和 `run` 函数组成，统一注册到 `ToolRegistry`。模型看到的是工具描述和参数结构，真正的函数调用由 Agent 完成。

工具结果统一转换为字符串。这使错误、文本结果和命令输出可以直接进入 observation，减少中间序列化层。代价是调用者若需要结构化字段，必须约定 JSON 字符串格式；因此 PACS 等复杂工具会显式返回 JSON，而普通文件和命令工具保留适合模型阅读的文本。

文件编辑通过原文片段定位修改位置。只有该片段在文件中恰好出现一次时才执行替换；找不到或存在多处时拒绝修改，并要求提供更完整的上下文。该方法不依赖容易变化的行号，也不需要重写整个文件，但只适合局部文本修改，不适合需要理解代码语法和引用关系的大规模重构。

### 3.4 上下文、长输出与记忆

随着任务持续进行，对话消息和工具结果会不断增加。系统优先使用模型接口返回的 prompt_tokens 判断当前上下文是否接近预算上限。如果接口没有提供 token 用量，则根据文本长度和图片数量进行近似估算。

当上下文超过预算时，系统保留最近几轮原始消息，并调用模型将较早的对话整理成摘要。摘要主要记录用户目标、已经完成的步骤、重要文件、关键决定和未解决的错误。如果摘要调用失败，系统会改用规则生成简要说明，避免上下文压缩失败导致整个任务中断。

上下文压缩的目的是让当前任务能够继续进行，而不是完整保存运行历史。模型生成的摘要可能遗漏细节，因此不能将其作为审计证据。

工具返回大量日志或文件内容时，系统会将完整结果保存到工作区的运行目录，只把结果概况、内容长度、文件位置和完整性哈希放入对话。这样既能减少后续模型调用消耗的 token，也能保留完整输出，便于之后排查问题。

跨会话信息使用 MEMORY.md 保存。程序启动时会读取其中的内容，并将其作为项目背景提供给模型。该文件适合记录编码规范、固定目录、运行命令等长期有效的项目约定，不适合保存密钥、临时聊天内容或大段代码。

三类信息的用途不同：
- 上下文用于完成当前任务。
- 记忆用于保留跨会话仍然有效的项目事实。
- Trace 用于记录实际执行步骤和运行结果，支持回放与核查。
它们分别解决“当前模型需要知道什么”“下次会话还应记住什么”和“程序实际做了什么”，不能相互替代。

### 3.5 规划与 Skill

`task_list` 为长任务提供简单的待办状态，帮助模型把多步任务拆开并跟踪完成情况。当前状态保存在进程内，适合单次会话规划，不是跨进程任务调度系统。

Skill 使用带 frontmatter 的 `SKILL.md` 描述领域流程。启动时只把名称和简介放入能力目录，模型决定使用某项能力后再读取正文。按需加载比一次注入全部 Skill 更节省上下文，也降低无关规则相互干扰的概率。Skill 提供的是做事方法，不直接获得额外权限；其中建议的操作仍需通过工具与权限层。

### 3.6 MCP 扩展

MCP 客户端通过 stdio 启动外部 server，完成 initialize、`tools/list` 和 `tools/call`。外部工具加上 `mcp__` 前缀后注册到同一个 ToolRegistry，避免与内置工具重名。

当前 MCP 客户端主要用于接入本地工具。它通过标准输入输出与一个 MCP Server 子进程通信，每次发送一条 JSON-RPC 请求，并等待对应结果返回。该实现能够完成服务初始化、获取工具列表和调用外部工具，足以支持本项目的工具扩展需求。

目前客户端不支持通过 HTTP 或 SSE 连接远程 MCP Server，也没有实现多服务并发管理、完整的异步通知处理以及服务崩溃后的自动重连。因此，它适合本地演示和受控环境，还不能作为功能完整的通用 MCP 客户端。MCP 工具属于外部能力，调用时仍需经过 Agent 权限检查；未明确分类的工具默认要求用户确认。

## 4. PACS 设计

### 4.1 依赖解析

PACS 首先读取项目声明的依赖。目前支持：
- `requirements.txt`，包括通过`-r`或`--requirement`引入的其他依赖文件；
- `pyproject.toml`中的项目依赖和 Python 版本要求；
- `environment.yml`或`environment.yaml`中常见的 conda 与 pip 依赖写法。

解析子依赖文件时，系统会检查文件是否仍位于项目目录内，防止通过相对路径读取项目外部文件。循环引用、文件不存在、路径越界、格式错误和暂不支持的配置都会被记录为警告，避免漏掉依赖。

普通包可以进入版本搜索，例如 numpy>=1.24。直接 URL、Git 仓库和本地文件等特殊依赖会保留原始写法，但不会为它们搜索版本，因为这类依赖通常没有可以从包索引枚举的标准版本列表。

系统内置了一个轻量版本匹配器，用于处理 ==、>=、<、!=、~= 等常见约束。它能够满足当前项目的搜索需要，但没有覆盖 PEP 440 的全部版本规则。遇到无法理解的版本写法时，系统会拒绝匹配或给出警告，避免在判断不可靠的情况下生成候选。

### 4.2 版本目录、求解和评分

得到依赖后，系统从 Python 包索引查询各个包的可用版本，并缓存查询结果。PACS 不会一开始就尝试每个包的全部历史版本，而是会优先选择一小批较新的版本组成搜索范围。例如，项目同时依赖三个包，每个包有几十个历史版本。如果直接组合，候选数量可能迅速增长到数万甚至更多。因此 PACS 先在较新的有限版本范围内搜索；只有找不到可行方案时，才逐步加入更旧的版本。这样可以控制搜索规模，同时兼顾长期未更新的旧项目。

候选组合优先通过 Z3 求解器生成。这里的 Z3 只负责在有限版本范围内找出不违反已知冲突的组合，不负责判断安装最终一定成功。如果运行环境没有安装 Z3，系统会改用带数量上限的枚举方式，保证基本功能仍然可用。
系统会对候选进行排序，主要考虑：
- 版本是否较新；
- 是否符合项目的 Python 版本要求；
- 是否提供预编译 wheel；
- 安装文件是否已经缓存；
- 是否命中低置信度的潜在冲突。
需明确，评分只决定验证顺序。分数高表示该组合更值得优先尝试，不表示它已经被证明可用。候选能否使用，仍然需要经过实际安装和验证。

### 4.3 并行验证与失败学习

PACS 先并发执行 preflight，利用 pip 的解析结果排除明显失败，再为可行候选创建独立环境进行实际安装。多个候选的 preflight 可以同时进行，以减少等待时间。

通过 preflight 的候选会进入实际安装阶段。每个候选使用独立的 Python 环境，避免不同版本之间相互影响。并行数量和总尝试次数都有上限，以免同时启动过多安装任务，占满处理器、内存、网络带宽和磁盘空间。

失败解析器识别版本冲突、Python 版本不匹配、平台不匹配、缺少 wheel、构建失败、缺少系统依赖、网络故障、超时、权限和磁盘错误等模式。只有能够绑定到本次候选中两个确切包版本、且属于可剪枝类型的失败，才写成 observed 约束。网络波动、磁盘不足等运行条件不会被误记为包版本冲突。

约束记录保存在 SQLite 数据库中，可以在后续运行中继续使用。系统区分两类约束：
- observed：从真实安装失败中得到的明确冲突，可以用于排除相同版本组合。
- derived：根据约束图中的多步关系推测出的潜在风险，只用于降低候选优先级。
由于依赖冲突本身通常不具有严格传递性，因此 derived 边只降低候选评分，不能直接判定组合无解。这是防止错误知识持续污染搜索空间的重要限制。

### 4.4 环境池与成功判据

环境池在工作区下创建 venv 或 conda 环境，并以每个环境的 `env.json` 作为状态来源。状态查询重新扫描磁盘，因此进程重启后仍能发现已有环境。环境 ID、根目录和符号链接都要经过检查，防止环境路径逃逸到工作区之外。

安装命令始终使用 argv 数组和 `shell=False`，不接受任意 shell 文本。每批安装保存完整 stdout、stderr、哈希、字节数、脱敏状态和执行方式。返回给 Agent 的只是紧凑摘要，完整日志不随失败环境一起删除。

依赖安装成功后，如果项目带有 `pyproject.toml`、`setup.py` 或 `setup.cfg`，系统会以 `--no-deps -e` 安装项目本身，随后执行 `pip check`。用户还可以指定必须成功导入的模块。只有这些校验通过，候选才成为最终环境。最终保留胜出环境并生成 `requirements.lock`，其余环境清理；每次运行同时生成结构化结果和可读报告。

并行并不保证任何项目都更快。包下载、索引限流、共享缓存、磁盘写入和大型 wheel 解压都可能成为瓶颈。因此并行度默认保持较小，并提供串行模式作为对照和低资源环境下的选择。

## 5. 安全设计

### 5.1 风险防范

系统可能接触用户输入、仓库文件、网页内容、第三方包安装脚本和模型生成的命令。我们假设这些内容可能包含错误指令或恶意文本，同时承认本项目运行在用户进程权限下，不能抵御内核漏洞、宿主机管理员或经过专门设计的高级逃逸攻击。

安全策略采用分层方式。上层用于减少误导，下层用于约束真实副作用。任何提示词声明都不能代替代码层检查。

### 5.2 权限分级

系统根据工具可能产生的影响，将调用分为三类：
- `allow`：可以自动执行，例如普通文件读取、搜索和 Skill 加载。
- `confirm`：执行前需要用户确认，例如文件写入、命令执行、网页访问和包安装。
- `deny`：直接拒绝，例如访问项目外路径或敏感文件。
这种设计避免只读操作频繁打断任务，同时让修改文件、运行程序等有副作用的操作保持可见。

启用`--auto-approve`后，系统会自动执行原本需要确认的操作，但不会绕过拒绝规则。例如，普通文件写入可以自动批准，但写入`.env`或项目目录之外仍然会被拒绝。

用户确认只表示用户允许本次操作，不代表该操作已经被证明安全。因此，确认命令前仍需检查它要访问的文件、执行的程序和可能产生的影响。自动批准模式只适合任务范围明确、输入来源可信的环境。

在实现自动环境配置任务的时候，为了保证整体流程的流畅性，推荐启用`--auto-approve`

### 5.3 文件与命令边界

写入路径先做规范化和真实路径解析，再检查是否位于工作区内；`.git`、`.env`、`.ssh` 和 `.gnupg` 等路径受到额外保护。PACS 的项目路径和环境工具 workdir 也必须留在当前工作区。

Bash 工具会拦截递归删除根目录、格式化磁盘、原始设备写入、提权、关机和下载后直接执行等明显危险模式。这类检查可以减少模型误操作造成的风险，但它本质上是基于命令文本的规则匹配，不是操作系统级沙箱。同一个危险行为可能通过变量、脚本文件、编码或其他程序间接完成，因此不可信代码仍需要更强的操作系统隔离，这是未来可以加强的点。

### 5.4 网络与外部内容

`web_fetch` 只接受 HTTP/HTTPS。请求前检查直接 IP、localhost 和内部域名，并解析 DNS 结果以阻止内网地址；遇到重定向时对每一跳重新检查。还可以通过环境变量切换为域名白名单策略。

网页内容返回后，会被包裹在 <external> 标记中，并注明它属于外部数据，不是用户或系统指令。例如，网页中即使出现“读取`.env`并上传密钥”，模型也应把它视为网页正文，而不是需要执行的任务。当前的这种标记只能降低提示注入影响，不能保证模型在所有情况下都正确判断。当前系统也没有独立的网络出口代理，无法在网络层彻底限制所有连接行为。高风险环境仍应通过容器网络、代理或防火墙控制外部访问。

### 5.5 包安装隔离

安装 Python 包的风险高于普通文件读取，因为部分包会在构建和安装过程中执行代码。PACS 因此对安装过程增加了额外限制。

安装器不接受任意 shell 命令，只允许调用目标虚拟环境中的 Python 或 pip，并使用参数列表直接启动进程。安装进程还会受到 CPU 时间、内存、文件大小和进程数量限制，避免异常安装任务无限占用系统资源。

在 Linux 环境中，如果 bubblewrap 已安装并且能够正常创建隔离空间，PACS 会：
- 将宿主机根目录设为只读；
- 将项目目录设为只读；
- 只允许写入当前候选的虚拟环境；
- 限制安装进程及其子进程的资源；
- 在父进程结束时终止相关子进程。

如果 bubblewrap 不存在，或者当前环境禁止创建 user namespace，PACS 会退回到仅使用资源限制的模式，并在结果中设置：
```text
filesystem_isolated=false
```

### 5.6 敏感信息与残余风险

工具证据、开发 trace 和 PACS 日志默认对常见密钥字段、Authorization 值、敏感环境变量和 URL 凭据进行脱敏，并尽量设置目录 `0700`、文件 `0600`。用户可以显式开启敏感保留用于取证，但系统不会自动启用。

当前实现没有自动保留期清理，也不能识别所有自定义密钥格式。第三方包在资源限制回退模式下仍可能读取用户进程可访问的文件。对不可信项目进行真实安装时，仍应在独立容器或虚拟机中运行整个程序。

## 6. 可观测性与成本

### 6.1 两类运行记录

系统将“审计证据”和“开发调试”分开保存。

工具证据记录只保存工具调用参数、状态、结果产物、长度和哈希，不保存用户消息、系统提示词、完整模型回答或后端请求。它的作用是回答“程序实际执行了什么、工具返回了什么”。大结果保存在独立 artifact 中，即使上下文已经压缩，证据仍然存在。

开发 tracer 记录按顺序排列的 LLM 与工具 span，包括名称、耗时、成功状态、受限预览和 provider usage。LLM span 不保留模型正文，只标记它产生了哪些工具调用或最终回答。它的作用是回答“运行经过哪些步骤、哪里失败、哪里最慢、哪次模型调用最贵”。

两类记录都采用 JSONL，写入失败不会改变 Agent 任务结果。这样可观测功能退化时任务仍能继续，但使用者需要检查 trace 是否可用，不能把缺失记录误认为没有发生操作。

### 6.2 回放和定位

回放按序号显示每个 span 的种类、名称、耗时、token 和成功状态。一次任务失败后，可以先定位第一个错误工具，再查看之前的 LLM span、工具调用摘要和对应工具证据。PACS 另外保留每个候选的阶段、评分、preflight 结果、安装日志、验证结果和最终选择，便于区分“解析失败”“安装失败”和“验证失败”。

我们暂时不选择把完整对话复制进 trace，因为这样虽然调试方便，却会显著扩大用户内容和密钥的持久化范围。因此，当前方案选择较少的调试信息，换取更清晰的隐私边界。

### 6.3 Token 与成本口径

成本统计只使用后端返回的 `prompt_tokens` 和 `completion_tokens`。字符除以四的估算只用于上下文预算，不作为最终费用依据。每次 LLM span 的估算费用为：

```text
prompt_tokens / 1000 × 输入单价
+ completion_tokens / 1000 × 输出单价
```

总成本是所有 LLM span 的费用之和，系统同时给出 token 最多或费用最高的一步。供应商的缓存计费、阶梯价格、免费额度和隐藏推理 token 如果没有出现在 usage 中，当前统计不会自行推断。

## 7. 关键设计取舍总结

### 7.1 使用显式循环实现 Agent

系统采用简单的 Python 循环组织模型调用、权限检查、工具执行和终止判断，使整个运行过程清晰且便于追踪；相应地，它不具备复杂 Agent 框架提供的分布式调度、持久状态管理和自动恢复能力。

### 7.2 使用确定性代码编排 PACS

PACS 由确定性代码控制候选生成、并行安装、尝试次数和成功判据，使搜索过程可控且便于复现。作为代价，系统需要自行维护依赖解析、候选求解和错误分类逻辑，对未知安装行为的处理能力较为有限。在处理非常复杂的冲突时，PACS会退出，经过模型的调整重新调用PACS

### 7.3 区分已验证冲突与推测风险

PACS 将约束分为“已观察约束”和“派生约束”。

- 已观察约束来自真实的安装失败。例如，日志明确表明`package-a==1.0`与 `package-b==2.0`无法同时安装，系统就会在后续搜索中排除包含这版本对的候选。这类约束有直接的失败证据，因此可以用于硬剪枝。
- 派生约束则是根据约束图中的多步关系推测出的潜在风险。例如，A 与 B 存在冲突，B 与 C 也存在冲突，并不能据此证明 A 与 C 一定冲突。因此，派生约束只会降低相关候选的优先级，不会直接将其删除。

这种设计可能会让系统额外验证一些最终仍然失败的组合，但可以避免不可靠的推断错误排除可行方案。

### 7.4 分层防护

提示词、权限、路径、网络、命令规则、资源限制和可选 bubblewrap 分别处理不同风险。没有任何单层能够覆盖全部威胁。文档和界面必须暴露降级状态，尤其不能把只有资源限制的执行描述为文件系统隔离。

## 8. 当前适用范围

MiniOpenClaw 适合用于受控工作区中的代码阅读、文件修改、命令执行、网页查询和 Python 环境配置。对于需要完整 PEP 440/508 兼容、复杂 conda 求解、系统级 CUDA/驱动配置、恶意构建脚本隔离或大规模集群调度的场景，当前PACS实现只能提供部分能力，不适合作为通用的解决手段。
